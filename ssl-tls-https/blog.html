SSL and TLS are encryption protocols used for security. For this blog post, I will be talking mostly about its integration with the web Hypertext Transfer Protocol (HTTP). By running HTTP over SSL/TLS, you enable an extra layer of security in what is commonly known as HTTPS. Since HTTP is an application layer protocol in the OSI layer model, SSL/TLS acts as a verification middle-man between the application and the server.

SSL/TLS

In 1993, Netscape introduced SSL, an encryption protocol that would sit between the TCP and Application layer. Its purpose was to ensure security between the client and the server, but the first version was actually never released because of its severe bugs and flaws. The first official release was the second version, released in 1995. Over time, even more bugs have been fixed, and has been put to rest in 2015. It is now deprecated, and IEFT released a 2015 statement announcing that everyone should use TLS instead.

There are currently five SSL/TLS protocols available. In short, the best practice is to support TLS v1.0 or higher. As of June 3, 2018, you can see we are working toward that goal!

How SSL/TLS Works

SSL was meant to be the answer to authentication and authorization problems on the web. Netscape believed that RSA encryption could take care of this, so the SSL handshake was born. A different key is created for each client and server connection.

Message digests were also introduced to detect and prevent data tampering. TLS does essentially the same process, with more secure underlying cryptographic algorithms.

SSL Security Issues

As usual, misconfigured SSL preconditions may lead to man-in-the-middle attacks. Usually, this means the server key has been stolen by the attacker, or the Certificate Authority has been spoofed/is untrustworthy/has been stolen. The SSL key exchange relies on weak algorithms, such as SHA-1 and MD5, which makes these attacks much more vulnerable. SSL is also pretty limited in security, especially compared to what TLS can offer.

The most famous weakness of SSL was an exploit dubbed 'POODLE', which took advantage of downgrading from TLS back to SSL. This attack triggered the end of SSL maintenance, which is super hilarious to me. Death by poodle.

You can test for POODLE by using testssl.sh, which I will also talk more about later. Luckily, only about 0.7% of the 138,874 websites surveyed by SSL Labs are still vulnerable to the POODLE attack, so we don't necessarily have to worry too much.

TLS Takeover

The Transport Layer Security protocol is the more secure successor of SSL. Its first version was released in 1999, as an upgrade to SSL v3.0. It has had a few updates since, with the most recent version proposed as of March 21, 2018. The Payment Card Industry Security Standards Council even suggests that everyone upgrades or migrates to TLS before June 30, 2018, which is just around the corner.

I already mentioned that SSL relied on weak algorithms. TLS v1.3 has removed support for MD5 and SHA-224 hash functions, and continues to push for even stronger options. We saw SSL's attempt to protect data with message digests, and now TLS also uses more secure key HMAC to avoid data tampering. The "Enhanced Pseudorandom Function" is also used to generate key data, using two hash algorithms to increase security.

Current Issues

There have been a couple of cryptographic exploits, which can be fixed by disabling weak SSL ciphers, and enabling strong TLS ones. There are also some other issues in RSA and other key negotiations. Luckily, some exploits, such as BEAST (Browser Exploit Against SSL/TLS) are no longer considered valid threats on most modern browsers.

SWEET32 / Birthday Attacks

I love Javascript. It's so versatile. Its powers can be used for good and evil. Essentially, with the SWEET32 vulnerability, an attacker can inject Javascript to override the "secure" flag on cookies. According to sap:

"With the SWEET32 vulnerability, it is now shown that an attacker can send in large volume of dummy data, and get blocks of cipher text that matches that of a customer. It is possible because the 64-bit blocks used by 3DES-DES generates a lot of packets, which increases the probability of an encrypted packet of a customer and that of the attacker matching."

This is definitely a man-in-the-middle attack, since the attacker intercepts the HTTPS connnection, especially during cross-site scripting.

Protocol Downgrade Attacks

Similar in logic to how the POODLE exploit worked, servers that support a TLS fallback signalling suite can detect protocol downgrade attacks by clients. It is important to enable support for this kind of defense. Another example of a downgrade attack is intercepting web traffic and redirecting users to an unencrypted HTTP version of the website. They are usually part of a larger man-in-the-middle attack, and can usually be fixed by removing backward compatibility.

Certificate Authority Exploits

A Certificate Authority (CA) is who you would buy your SSL/TLS certificate from. They have a private key to sign the issued Root Certificate, which is a public key, trusted by browsers to verify valid HTTPS websites. Some popular CA authorities include Comodo, Digicert, and Symantec. You can get different types of SSL/TLS certificates depending on what you need in validation or domain setup. The "rule of thumb" is to check your browser for a secure status, usually indicated by a green lock and the https:// protocol.

In 2011, which some security experts deem to be the worst year for CAs, there were a handful of digital certificate attacks. Namely, there were was a compromise with Comodo, which breached nine certificates across seven domains, including login.yahoo.com, mail.google.com, login.skype.com, and addons.mozilla.org, which are all typically trusted websites. Luckily, this attack was found pretty quickly, and the certificates were quickly revoked.

Other CA attacks may include (more) man-in-the-middle attacks that play with digital certificates and SSL/TLS traffic. Most of these can be prevented by using stricter browser controls. There have also been problems with illegitimate certificates installed by malware, or mistakes made when a CA sells to an illegitimate company. Infosec Institute has a really nice breakdown of other ways Certificate Authorities have been exploited, and even how you can steal your own digital certificate. Luckily, there are a lot more measures being taken on the CA side of things to prevent similar exploits.

Active Network Attacks

Moxie Marlinspike introduced an SSL "stripping" attack in 2009, and has even offered a tool to exploit it called sslstrip. The project looks like it has been untouched for the last seven years. On the official software page, moxie describes the exploit as such:

"It will transparently hijack HTTP traffic on a network, watch for HTTPS links and redirects, then map those links into either look-alike HTTP links or homograph-similar HTTPS links. It also supports modes for supplying a favicon which looks like a lock icon, selective logging, and session denial."

There are a couple of other active network attacks, such as upside-down-ternet and dsniff. Upside-down-ternet is more of a prank you can play on your friends, but it does showcase the vulnerabilities available in this space.

Implementing HTTPS

HTTPS is any HTTP request encrypted over SSL/TLS as an extra step to protect traffic. We're looking at a huge push for HTTPS right now, with even browsers such as Chrome advocating for HTTPS, or else they'll warn visitors or even block people from entering your site. According to statoperator, about 355,216 out of the top 1,000,000 websites sites use HTTPS as a default, with the number climbing even more every day (as of June 7, 2018).

Free Options

With the movement to push all web pages to serve HTTPS, some projects offer free or shared certificates. The most popular one is Let's Encrypt, but the service I have the most experience with is Cloudflare. However, more companies are coming forward with different options, and I hope soon, HTTP can be deprecated on the web altogether. Might be a bit of a dream, but I think it could be possible.

Issues with HTTPS

Absolutely all pages should be served over HTTPS. Failure to server sources such as CSS, scripts, images, and AJAX requests allows room for man-in-the-middle attacks. This means we have to be super careful about the Mixed Content issue. It only takes one HTTP request to reach HTTPS pages. Also remember to preload the HTTP Strict Transport Security header into browsers, and make sure all of your cookies are marked as secure. Misconfiguration is always going to be an issue.

To temporarily solve mixed-content, put this meta tag into each of your head tags:

<meta http-equiv="Content-Security-Policy" content="upgrade-insecure-requests">

This will treat all of your insecure HTTP URLs as though they are served as HTTPS. This is only a temporary solution in a lot of cases (some are generally fine, like images hosted over HTTPS anyway) because if you try to access a resource over HTTPS that is not actually served with SSL/TLS, you will not get a response. If you use the upgrade-insecure-requests content security policy (CSP), be sure you do not implement block-all-mixed-content.

HSTS

The HTTP Strict Transport Security policy protects against downgrade attacks and cookie hijacking, as we've previously discussed. It does this by setting strict declarations of which connections can be accepted, and does not run over insecure HTTP. Essentially, the server asks the browser via an HTTPS response if there is a Strict-Transport-Security policy in place, and if the header is valid, it will allow the browser secure access for a limited amount of time.

This solves the issue of many man-in-the-middle attacks, since it ensures that the connection is and should be secure. It also protects against KRACK! HSTS should definitely be pre-loaded, since the HSTS header can be stripped by an attacker if it is their first visit. To enable preloading, you can add a preload token to your server's response header, and submit your domain to a maintained list of preloaded sites.

SSL/TLS/HTTPS Maintenance

I found this cool guide on SSL/TLS security testing with testssl.sh. It covers several bases, including SWEET32/FREAK/POODLE/etc testing, telling you what ciphers your server supports, and giving you warnings. A full list of features can be found on the testssl git repo. If you're into front-end interfaces, TKCERT also created an app for that!

I already talked about content-security-policy, but to really take advantage of it, please enable CSP Reports. They are (weirdly) not sent by default, so you need to specify the policy directive and a URI for delivery.

You can also go through OWASP's thorough SSL/TLS testing guide.
